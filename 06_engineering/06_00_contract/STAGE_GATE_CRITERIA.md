# Stage Gate Criteria

| Stage | Goal | Required Artifacts | Validation Method | Exit Criteria | Forbidden Premature Moves |
| --- | --- | --- | --- | --- | --- |
| Stage 0 | Freeze scope, assumptions, repo discipline, and reporting scaffold. | `SYSTEM_SCOPE.md`, `HARDWARE_ASSUMPTIONS.md`, `REPO_EXECUTION_POLICY.md`, `STAGE_GATE_CRITERIA.md`, `MASTER_EXECUTION_REPORT.md`, `STAGE_00_REPORT.md`, scaffold directories, placeholder READMEs. | File presence check, branch check, commit check, manual review of scope constraints. | Stage 0 documents exist, reports exist, branch is correct, commit is made, no implementation artifacts for later stages were created prematurely. | Designing MQTT contract in detail, writing edge daemon, implementing backend/UI, building simulation model, writing next-stage tests. |
| Stage 1 | Freeze minimal system contract and simulation use-case boundaries. | Interface boundary notes, scenario list, system context diagram, MVP message boundary draft, Stage 1 report. | Manual review against Stage 0 scope, traceability to MVP scenarios. | Minimal architecture and interface boundaries are explicit enough to implement without scope ambiguity. | Hardware-specific coding, backend implementation, UI implementation, ROS 2 adoption. |
| Stage 2 | Prepare functional software-only twin against the frozen contract. | Twin scaffold, simulation README, reproducible local run instructions, scenario evidence, Stage 2 report. | Reproducible local scenario execution. | Functional twin runs the approved contract scenarios locally without making Webots or hardware claims. | Real hardware claims, backend expansion beyond MVP needs, presenting Webots as implemented without evidence. |
| Stage 3 | Implement edge-side MVP logic against simulation interfaces. | Edge package scaffold, adapter boundary, command/state flow implementation, run instructions, Stage 3 report. | Local run validation with simulation-side interaction and logs. | Edge service can exchange bounded commands/status within approved MVP path. | Hardware-specific assumptions presented as verified fact, ROS 2, advanced autonomy features. |
| Stage 4 | Implement MQTT transport contract and broker-backed message flow. | MQTT topic contract, transport notes, local broker-backed evidence logs, Stage 4 report. | End-to-end message exchange validation with reproducible commands/logs. | MQTT flow is evidenced between edge and server-side components in a local demo contour. | Fleet-scale features, overcomplicated QoS matrix without evidence need, claiming deployment-grade Mosquitto validation. |
| Stage 5 | Implement backend and telemetry persistence MVP. | FastAPI service, storage boundary, dev/demo storage path or equivalent, storage README, Stage 5 report. | API health validation, telemetry write/read validation, reproducible service startup. | Backend stores and exposes required MVP telemetry/state in an honestly described MVP path. | Analytics overbuild, non-MVP microservices, claiming PostgreSQL as current evidenced runtime if only abstraction/dev path exists. |
| Stage 6 | Implement operator-facing MVP view/control path. | Operator UI or thin control surface, run instructions, validation evidence, Stage 6 report. | Manual operator flow validation against MVP scenarios. | Operator can observe state and trigger approved supervised control action path. | Rich dashboard expansion, fleet features, non-MVP UX work. |
| Stage 7 | Execute integration testing and collect evidence. | Test procedures, logs, screenshots, issue list, evidence index, Stage 7 report. | Repeatable end-to-end runs in simulation-first setup. | MVP behavior is reproducibly demonstrated with evidence sufficient for VKR implementation claims. | Claims of hardware validation without hardware evidence, premature thesis conclusions. |
| Stage 8 | Hardware-aligned demonstration and VKR evidence consolidation. | Hardware validation notes if hardware is available, final evidence mapping, limitations list, Stage 8 report. | Supervisor/lab-aligned demonstration review and evidence check. | Final demonstrable contour and honest limitations are documented for VKR without overstating software-only proof as hardware readiness. | Claiming industrial readiness, certified safety, or unsupported hardware functionality. |
